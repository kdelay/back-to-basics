# 클러스터형 인덱스와 MVCC
> Linear: LIL-198 | Phase 5 | 2026-07-23 | (QnA 학습, 강의 없음)
> → velog: back-to-basics-03 (붙여둘까 떼어둘까, 그 결정의 뿌리엔 MVCC가 있었다)

## Q1. 클러스터형 인덱스랑 넌클러스터형, 뭐가 달라? cluster라는 단어부터 안 와닿아.
**A.** cluster = "관련 있는 것끼리 가까이 모아둔 무리". 클러스터형 인덱스는 인덱스 순서대로 실제 데이터를 물리적으로 붙여 저장한다(PK 순으로 행이 줄줄이). 넌클러스터형은 데이터는 흩어놓고 "어디 있는지 주소(PK)"만 정렬해 둔다(책 뒤 찾아보기처럼).

## Q2. 클러스터형은 데이터가 붙어 있다면서, 왜 어떤 조회는 PK로 또 찾아? 그럼 넌클러스터 아냐?
**A.** 클러스터형 테이블에도 인덱스는 여러 개다. 데이터가 붙은 건 PK 인덱스 하나뿐. email 같은 세컨더리 인덱스는 데이터를 복사하면 낭비라 PK 값만 담는다. 그래서 email로 찾으면 → email 인덱스에서 PK를 얻고 → PK 인덱스로 데이터를 한 번 더 방문(두 번 탄다). 결국 세컨더리 인덱스는 사실상 넌클러스터형처럼 동작.

**💡 왜/트레이드오프:** 데이터를 한 벌만 유지(저장 절약, 수정 시 한 곳만) vs 세컨더리 조회는 2단계.

## Q3. 그럼 클러스터형/넌클러스터형은 뭐랑 매핑되는 개념이야? 왜 테이블당 하나만 클러스터형?
**A.** 데이터 행의 "물리적 저장 순서"와 매핑. 디스크엔 한 가지 순서로만 저장 가능(id순이면서 동시에 email순일 수 없음). 그 물리 순서를 정하는 인덱스가 클러스터형(하나뿐), 나머지는 넌클러스터형.

## Q4. MySQL은 클러스터형, PostgreSQL은 넌클러스터형이라는데 왜 정반대를 택했어?
**A.** 뿌리는 MVCC(여러 버전을 유지해 동시 읽기/쓰기 충돌 방지). 갈림길은 "수정 시 옛 버전을 어디 두나".
- **PostgreSQL**: 새 버전을 새 위치에 통째로 씀 → 행 위치가 계속 바뀜 → 물리 정렬 불가 → heap(넌클러스터형), 죽은 버전은 VACUUM으로 청소.
- **MySQL InnoDB**: 제자리 수정 + 옛 버전은 undo log에 보관 → 행 위치 유지 → PK 순 정렬 유지 → 클러스터형 가능. PK 조회가 압도적인 웹서비스에 유리.

**💡 왜/트레이드오프:** InnoDB는 PK 조회가 한 번에 끝(웹 최적) / PG는 갱신 많을 때 유리하나 VACUUM 필요. **UUID를 PK로 쓰면 안 되는 이유**도 여기서 나온다(랜덤 PK → 클러스터 정렬이 계속 깨져 페이지 분할이 잦아짐).

## 정석 순서 (거꾸로 배운 걸 정리)
저장(행이 페이지에 쌓임) → 인덱스(색인) → MVCC(동시성) → 그 결과로 나오는 저장 구조(클러스터형 vs 넌클러스터형). 질문은 결과(4번)부터 했지만, 거꾸로 파고들며 뿌리(MVCC)까지 닿았다. → 정석 순서 학습은 LIL-200에서.
