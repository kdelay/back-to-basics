# 스프링 웹 개발 3가지 방식 — 정적 컨텐츠·MVC·API
> Linear: LIL-165 | Phase 1 | 2026-07-23 | 스프링 입문 섹션 3
> ※ 대화 원본 유실 후 진행로그에서 역구성. 이후 세션부터 실시간 축적.

## Q1. 스프링에서 웹 요청을 처리하는 방식이 여러 개야?
**A.** 크게 3가지.
1. **정적 컨텐츠**: `static/` 폴더 파일을 가공 없이 그대로 반환.
2. **MVC/템플릿엔진**: 컨트롤러가 데이터를 `Model`에 담고, 뷰 리졸버가 `templates/*.html`을 찾아 Thymeleaf가 렌더링해 완성 HTML 반환.
3. **API**: `@ResponseBody`로 뷰가 아니라 데이터(JSON)를 직접 반환.

## Q2. 정적 파일이랑 컨트롤러 경로가 겹치면 뭐가 이겨?
**A.** 컨트롤러 매핑이 우선. 스프링이 요청 URL에 매핑된 컨트롤러를 먼저 찾고, 없으면 그때 static 리소스를 본다.

**💡 왜/트레이드오프:** 동적 처리(컨트롤러)에 우선권을 줘서 같은 경로라도 로직으로 가로챌 수 있다.

## Q3. static이랑 templates는 왜 폴더가 나뉘어 있어?
**A.** static은 "완제품"(그대로 내보냄), templates는 "반제품"(서버가 데이터를 끼워 렌더링해야 완성). templates를 직접 URL로 접근 못 하게 분리한 것.

## Q4. @ResponseBody를 붙이면 뭐가 달라져?
**A.** 뷰 리졸버를 거치지 않고 HTTP 메시지 컨버터(객체면 Jackson)가 반환값을 응답 body에 직접 쓴다. 객체 → JSON 직렬화. `@RestController`는 `@Controller + @ResponseBody`.

**💡 왜/트레이드오프:** `@ResponseBody`(응답 방향) vs `@RequestBody`(요청 body를 객체로, 반대 방향). 화면이 필요 없고 데이터만 주고받을 때(앱·SPA) API 방식.

## "왜?" 트레이드오프 (섹션 3)
왜 요즘은 템플릿 엔진보다 API(JSON)로 수렴하나? → 웹·앱 등 여러 클라이언트가 하나의 백엔드를 공유할 때, 화면까지 서버가 그리면 클라이언트별 컨트롤러를 중복 관리해야 한다. 데이터(JSON)만 주면 화면은 각 클라이언트가 그리니 백엔드 하나로 끝난다.
